# Reproduction notes

What was actually run, on what, and how to repeat it. This is a constrained
reproduction of the TRM recipe — not a clean-room reproduction of the paper's
numbers. See `docs/m1_limitations.md` for the hardware story.

## Environment

- Apple M1, 8 GB unified memory (`MacBookPro17,1`)
- Python 3.12, PyTorch 2.7 (MPS backend), NumPy, Matplotlib, Pillow
- No GPU beyond the M1's integrated MPS device

```bash
pip install torch numpy matplotlib pillow
```

## Runs that completed

| run | command | params | data | result | logged at |
| --- | ------- | ------ | ---- | ------ | --------- |
| 4×4 | `python3 build.py`     | 354K  | ~10K (200 base × 50 aug)   | **91.77%** val (epoch 37, 60 epochs) | `logs/trm_4x4_baseline/` |
| 6×6 | `python3 train_6x6.py` | 3.15M | 500K (1000 base × 500 aug) | **43.6%** val (~290 epochs)          | `logs/trm_6x6_baseline/` |

Both completed on the 8 GB M1. 6×6 requires the per-step backward loop in
`train_step_by_step.py`; the vanilla loop in `build.py` is for 4×4.

Checkpoints (`*.pt`) and generated datasets (`*.npy`) are gitignored — they are
regenerated by the commands above. The two `val_*_6x6_10k.npy` arrays are kept
so the 6×6 solve figures reproduce without regenerating data.

## Run that did not complete

| run | command | outcome |
| --- | ------- | ------- |
| 9×9 | `python3 train_9x9.py` | OOM on 8 GB M1; only `config.json` was written. See `results/9x9_failure_notes.md`. |

## Data generation

- `data_generation.py` — 4×4 puzzles + solutions
- `sudoku_6x6_generator.py` — 6×6 with value-permutation / row-swap / column-swap augmentations
- `sudoku_9x9_generator.py` — 9×9 (generation is slow, so the default 9×9 dataset is small)

The 6×6 dataset is built lazily inside `train_6x6.py`. Generating the full 500K
augmented 6×6 set takes on the order of tens of minutes on the M1.

## Regenerating the figures

All README/results figures rebuild from the committed logs and checkpoints:

```bash
python3 scripts/build_assets.py          # assets/ (curves, per-step, architecture, solves)
python3 scripts/build_success_asset.py   # results/small_sudoku_success.png
```

`build_assets.py` runs the solve figures on CPU and reads `logs/*/history.json`
+ `step_accuracies.json`; it skips any figure whose checkpoint is missing.

## Honest scope

- The 4×4 result is a genuine reproduction of the recipe's intended behavior on
  a laptop.
- The 6×6 result is a partial success and a useful negative finding (see the
  README's *Where it breaks down* section).
- The 9×9 result is a documented hardware wall, not a model result.

What would change with more compute: completing 9×9, training the Transformer
backbone variant (`use_attention=True`, implemented but untrained at scale) on
6×6, and generating a 9×9 dataset large enough for a ~2M-param model.
